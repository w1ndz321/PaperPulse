    花了147万美元，造出4.5万个可执行SWE环境：OpenSWE把“训练修Bug智能体”这件事彻底开源了

    TL;DR

    训练软件工程智能体，最大难题不是模型，而是缺少大规模、能跑起来、能验对错的真实环境。OpenSWE用一个多智能体流水线，在64台机器上自动合成了45,320个可执行Docker环境，覆盖12.8k个Python仓库，并用“难度感知筛选”剔除无解和太简单的数据。最终训出的32B/72B模型在SWE-Bench Verified上做到62.4%/66.0%，明显超过同类开源方案。

    软件工程智能体、环境合成、Docker可执行环境、多智能体流水线、难度感知筛选、SWE-Bench

    为什么SWE智能体难训练？卡住大家的不是模型，而是“真环境”太贵太难做

    现有痛点：数据不只是少，而是“不透明、不可复现、不够难”

    这篇论文盯上的核心问题很实际：SWE智能体不是写一段静态代码就结束了，它得像程序员一样进仓库、改代码、跑测试、看报错、再修一轮。

    问题在于，这种训练需要可执行、可验证、可反复交互的环境。也就是每条样本都得配好仓库快照、依赖、Docker、测试脚本，还要能判断“修复到底是不是真的有效”。

    作者指出，现有开源方案有几个硬伤：

    1. 规模不够大：仓库数量和任务数量都有限
    2. 仓库多样性不够：容易让模型学窄了
    3. 工业方案不透明：效果不错，但基础设施不公开
    4. 数据质量参差不齐：很多任务其实没法学

    两类最坑的数据：无解题和送分题

    论文特别强调了两种低价值环境：

    • PR-Issue misalignment（PR和Issue不对齐）：Issue说要修A，但PR真正修的是B，模型再聪明也学不会
    • Triviality（过于简单）：Issue里直接把答案说出来了，等于开卷抄答案

    作者给了两个很具体的例子：

    • 一个任务里，Issue要求检查完整40位commit hash，但真实补丁允许前7位短hash也通过测试，这就造成“描述和判题不一致”，本质上接近无解
    • 另一个任务里，Issue几乎直接写明：某个文件某一行的ra应该改成raise，这种样本几乎没有训练价值

    所以，论文的观点很明确：SWE训练数据不是越多越好，而是要先能跑，再够真，还得难度合适。

    OpenSWE到底造了多大一个场：4.5万个环境、1.28万个仓库、两周跑完

    规模上，OpenSWE是目前最大的全透明Python SWE训练框架

    OpenSWE给出的数字非常夸张，而且是全开源的：

    • 45,320 个可执行Docker环境
    • 覆盖 12.8k 个代码仓库
    • 基于约 572,114 个GitHub PR做筛选和构建
    • 在 64节点集群 上并行合成
    • 整个环境构建耗资 $891K
    • 后续轨迹采样和难度筛选又花了 $576K
    • 总投入约 $1.47M

    最后保留下来的是：

    • 约 9,000 个质量保证环境
    • 约 13,000 条精选训练轨迹

    这篇论文最有价值的一点，不只是“我做得大”，而是把Dockerfile、评测脚本、合成流水线、基础设施全放出来了。这对学术界特别重要——以前很多组不是不想做，而是根本搭不起这套系统。

    和已有数据集比，OpenSWE的“仓库覆盖”很猛

    论文表2给了直接对比。几个关键数字：

    • SWE-rebench：3.5k repos，21.3k tasks
    • SWE-rebench-v2（Python）：573 repos，7.2k tasks
    • Scale-SWE（开源部分）：1.2k repos，20.2k tasks
    • OpenSWE：12.8k repos，45.3k tasks

    如果只看“开源且真实仓库”，OpenSWE在仓库数量和可执行任务数上都拉开了明显差距。可以把它理解成：以前大家是在几个小区里练修Bug，现在作者直接开出了一座城市。

    多智能体流水线怎么造环境：从翻仓库、写Docker，到自动验真假

    第一步：先从GitHub PR里筛出“像样的题”

    作者先收集GitHub PR，再做四层过滤，只保留适合做SWE任务的样本：

    1. 仓库至少5个star：避免太新的、没人维护的项目
    2. 主语言必须是Python
    3. PR必须关联有描述的Issue
    4. 补丁必须改到非测试代码：只改tests/、spec/、e2e/这种文件的不要

    这一步的目标很朴素：先把“连题目都说不清”或者“只是在补测试”的样本扔掉。

    第二步：用4类Agent分工合作

    整个OpenSWE流水线像一个小型“软件工程外包团队”，核心有几类Agent：

    1. 仓库探索Agent
      不会把仓库整棵树都翻烂，而是重点找README、CONTRIBUTING、依赖文件、CI配置这些“高价值线索”。

    2. Dockerfile构建Agent
      负责给每个任务搭出能跑的环境。

    3. 评测脚本Agent
      负责生成bash脚本，确保测试能准确判断“补丁是否真的修好问题”。

    4. 测试分析Agent
      测试失败时，判断到底是Docker环境错了、评测脚本错了，还是这个任务本身无解。

    这套设计的关键不只是“自动化”，而是可迭代反馈：失败后不是全部推倒重来，而是把错误反馈给对应Agent继续修。

    为了大规模构建，作者做了很多很工程的优化

    这部分特别像“踩坑总结”，很接地气。

    预构建Python基础镜像，避免网络下载把流程拖死

    作者没有从通用Ubuntu镜像开始现装Python，而是预先做好一套 openswe-python 基础镜像，覆盖：

    • Python 2.7
    • Python 3.5 到 3.14

    这样做的好处很直接：

    • 避免构建时联网装Python导致超时
    • 共享底层镜像缓存，减少重复构建

    不在容器里git clone，直接本地缓存仓库

    他们维护本地bare repo缓存，构建时直接COPY代码进去，并提前checkout到目标commit。这样可以：

    • 避开GitHub API限速
    • 降低网络失败
    • 提高复现性

    镜像不乱删，脚本改了就复用旧镜像

    在常见情况下，如果只是评测脚本要改，而Dockerfile没变，他们会保留已有镜像继续用。这个优化带来了5×加速。

    最后怎么判断一个环境真的“有效”？

    论文采用双条件验证：

    1. 先只打测试补丁，验证原始代码上测试确实会失败
    2. 再打完整修复补丁，验证所有相关测试都能通过

    只有“先失败、后通过”同时满足，样本才算有效。

    另外，脚本里会输出一个明确标记：OPENSWE EXIT CODE=X，系统通过正则解析这个结果，避免日志乱七八糟时误判。

    为了稳住大规模运行，每个容器还限制资源：

    • 4个CPU核心
    • 24 GB内存
    • 200 GB存储

    不是所有环境都值得学：作者专门做了“难度感知筛选”

    他们不迷信数据量，而是筛“最有学习价值”的那一段

    论文一个很重要的观点是：只拼数量，训练效果不一定最好。因为真实仓库合成出来的环境里，很多要么无解，要么太简单。

    所以作者额外花了 $576K 做轨迹采样和筛选，目标是找出：

    • 不是无解题
    • 也不是一眼看穿的送分题
    • 难度刚好能提供学习信号

    最终从4.5万个环境里，精选出大约：

    • 9,000 个高质量环境
    • 13,000 条训练轨迹

    这有点像刷题：不是题库越大越好，而是要挑那些“做完能涨能力”的题。

    数据规模和质量是互补关系，不是二选一

    作者后面的实验很清楚地说明：

    • 更多高质量环境，性能会继续涨
    • 只堆原始环境，不做筛选，效率不如“难度合适的数据”
    • 所以真正有效的是：规模化构建 + 质量感知筛选

    这也是论文比很多“我们又造了个更大的数据集”更进一步的地方。

    结果有多强：32B做到62.4%，72B做到66.0%，而且越加数据越没到头

    在SWE-Bench Verified上，OpenSWE训练出的模型拿到强结果

    作者使用Qwen2.5-32B-Base和Qwen2.5-72B-Base做SFT，最终结果如下：

    • OpenSWE-32B + SWE-Agent：62.4%
    • OpenSWE-72B + SWE-Agent：66.0%
    • OpenSWE-32B + OpenHands：59.8%
    • OpenSWE-72B + OpenHands：65.0%

    几个关键对比：

    • 相比 daVinci-Dev-32B 的 56.1%，OpenSWE-32B 提高 6.3 个点
    • 相比 daVinci-Dev-72B 的 58.5%，OpenSWE-72B 提高 7.5 个点
    • 相比 SWE-Master-32B-RL 的 61.4%，OpenSWE-32B 也高 1.0 个点
    • 论文还特别强调：32B结果比强基线高 4.6%，而且它用的还不是Coder版底座

    OpenSWE比SWE-rebench训练数据更有效

    表4很有说服力。同样的模型和脚手架，换训练数据源后结果差别很大：

    SWE-Agent脚手架

    • 32B：
    • SWE-rebench：50.2%
    • OpenSWE：62.4%
    • 提升 12.2 个点
    • 72B：
    • SWE-rebench：63.4%
    • OpenSWE：66.0%
    • 提升 2.6 个点

    CodeAct / OpenHands脚手架

    • 32B：51.4% → 59.8%
    • 72B：62.4% → 65.0%

    这说明OpenSWE的优势不是某个Agent框架“调得巧”，而是训练环境本身更有用。

    数据继续加，性能还在稳定涨，没有看到饱和

    论文做了数据扩展分析，发现四条曲线都接近log-linear增长，也就是数据越多，性能越高，而且还没撞到天花板。

    相关系数分别是：

    • 72B + OpenHands：r=0.972
    • 72B + SWE-Agent：r=0.911
    • 32B + SWE-Agent：r=0.893
    • 32B + OpenHands：r=0.882

    这些数字说明：这种增长趋势很稳定，不是偶然波动。

    更有意思的是，大模型更吃数据：

    • OpenSWE-72B 比 OpenSWE-32B 最终高 3.6 个点
    • 而daVinci-Dev从32B扩到72B，只提升 2.4 个点

    换句话说，更好的训练环境，能让更大的模型把参数优势真正吃出来。

    不只会修Bug：代码、数学、科学题也一起涨了

    代码能力提升最明显

    OpenSWE训练后，代码基准涨幅非常大：

    32B

    • HumanEval：61.43 → 90.52，提升 +29.09
    • HumanEval+：54.01 → 85.24，提升 +31.23

    72B

    • HumanEval：66.82 → 76.25，提升 +9.43
    • HumanEval+：59.23 → 70.75，提升 +11.52

    这个不难理解：SWE任务本来就要求读代码、改代码、调代码，和代码生成能力高度重合。

    数学和科学推理也有外溢收益

    比较亮眼的是，SWE训练还提升了数学与科学推理：

    数学

    • 32B GSM8K：+6.14
    • 32B MATH-500：+8.20
    • 72B GSM8K：+5.99
    • 72B MATH-500：+12.20

    科学

    • 32B SuperGPQA：+5.77
    • 72B SuperGPQA：+8.10
    • SciBench分别提升 +4.80 和 +4.70

    作者的解释也挺合理：修Bug训练的其实是多步规划、排错、迭代验证，这些能力会迁移到数学推理和科学问答。

    但它不会让模型“背更多冷知识”

    在偏事实记忆的基准上，变化就小很多：

    • 32B MMLU：83.57 → 83.57，几乎不变
    • TriviaQA：提升也只有 +1.41 和 +2.85

    这说明OpenSWE主要增强的是过程性推理能力，不是给模型补百科知识库。

    总结：这篇论文最重要的，不是又刷高了分，而是把“SWE训练基础设施”公开了

    这项工作的真正价值

    OpenSWE最值得关注的地方，我觉得有三点：

    1. 把SWE训练从“少数大厂的黑箱能力”拉回开源社区
      不只是放结果，而是把环境、脚本、流水线都公开

    2. 证明高质量环境数据本身就是核心能力
      好数据不只是锦上添花，甚至能弥补底座不是专门Coder模型的劣势

    3. 给出一条明确路线：继续扩环境，性能还会涨
      论文已经看到稳定的log-linear增长，而且还没饱和

    一句话评价

    如果说以前大家都知道“SWE智能体要靠真环境练”，那这篇论文做的是：第一次把这套“真环境工厂”大规模、透明地搭出来了。
