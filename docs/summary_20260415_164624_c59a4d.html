    DETAILED ANALYSIS

    ACE：别再把提示词越改越短了，这篇论文让上下文像“作战手册”一样越用越强

    TL;DR

    很多上下文优化方法都有两个老毛病：一是越改越短，把关键领域经验压缩没了；二是越改越塌，一次重写就把几万 token 的知识缩水成几句话。Stanford、SambaNova 和伯克利提出的 ACE，把上下文当成持续生长的“作战手册”，通过生成、反思、整理三步持续增量更新。结果上，ACE 在 agent 任务平均提升 10.6%，金融任务提升 8.6%，适配延迟平均降低 86.9%，还能只靠执行反馈自我改进。

    为什么现有方法会“越优化越笨”：短小偏见和上下文坍塌

    1. 论文先点名两个核心问题

    作者认为，很多 LLM 应用现在不一定靠微调权重，而是靠改上下文：系统提示词、记忆、检索证据、策略说明等。但现有方法有两个明显缺陷：

    1. brevity bias（短小偏见）
      方法喜欢把提示词优化成简短、通用的版本，看起来更“优雅”，但会丢掉真正有用的领域细节，比如工具使用诀窍、常见坑、任务特定启发式。

    2. context collapse（上下文坍塌）
      如果每轮都让模型“整段重写”已有上下文，最后常会越写越短、越写越空，像把一本操作手册不断总结成一句口号。

    2. 这个“坍塌”有多严重？

    论文在 AppWorld 上给了一个很直观的例子：

    • 第 60 步时，上下文长度还有 18,282 tokens
    • 对应准确率 66.7%
    • 下一步重写后，直接塌成 122 tokens
    • 准确率跌到 57.1%
    • 甚至还不如没有上下文适配时的 63.7%

    这个现象很像你把团队经验文档交给一个实习生反复“帮忙精简”，最后只剩一句：“注意质量、认真执行。” 听起来没错，但几乎没法用。

    ACE 怎么做：把上下文从“一整段提示词”改成“持续进化的作战手册”

    1. 三个角色分工，不让一个模型包办所有事

    ACE 的核心思路很简单：别再让模型一次性重写整份上下文，而是把上下文拆成结构化条目，像维护 wiki 或作战手册一样慢慢长。

    它有三个模块：

    • Generator：先去做题、跑轨迹、调用工具，产生推理过程
    • Reflector：分析哪些地方做对了、哪些地方翻车了，提炼经验
    • Curator：把经验整理成小的上下文增量，再并入已有上下文

    这其实是在模拟一个靠谱团队的工作流：
    先实战，再复盘，最后沉淀 SOP。

    2. “增量更新”是 ACE 的关键

    ACE 不把上下文当成一整块大 prompt，而是拆成一个个 bullet 条目。每个条目包含：

    • 一个唯一 ID
    • 有帮助/有害的计数
    • 具体内容，比如某条策略、某个金融概念、某类常见错误

    新任务来了，模型会标记哪些条目有用、哪些误导，然后 Reflector 提出修订建议，Curator 只更新相关条目。

    这带来三个好处：

    • 局部修改：只改相关知识，不动整本“手册”
    • 细粒度检索：只拿最相关的条目给模型看
    • 持续积累：新经验可以不断追加，不会一把梭全重写

    3. Grow-and-Refine：不是无脑变长，而是边长边整理

    ACE 不是一味堆 token。它有个机制叫 grow-and-refine：

    • 新经验先加进去
    • 已有条目按反馈更新
    • 再做去重和冗余清理
    • 可以每次都整理，也可以等快超上下文窗口时再整理

    所以 ACE 的目标不是“更短”，也不是“更长”，而是更完整、更有组织、还能持续维护。

    在 AppWorld 上，ACE 用开源模型追平榜首商业系统

    1. 离线适配：平均分从 42.4% 拉到 59.4%

    在 AppWorld agent 基准上，基础模型 DeepSeek-V3.1 + ReAct 的平均成绩是 42.4%。

    离线适配后：

    • ReAct + ICL：46.0%
    • ReAct + GEPA：46.4%
    • ReAct + ACE：59.4%

    也就是说，ACE 相比：

    • ICL 高 13.4 个点
    • GEPA 高 13.0 个点
    • 比基线高 17.0 个点

    论文正文总结为：ACE 在 agent 任务上相对强基线平均提升 10.6%。

    2. 没有标注也能学：只靠执行反馈，依然提升 14.8%

    更有意思的是，ACE 不一定需要 ground-truth label。

    在没有 GT labels 的情况下：

    • ReAct 基线：42.4%
    • ReAct + ACE：57.2%

    提升了 14.8%

    原因是 agent 场景自带“天然反馈”：

    • 代码执行成功没
    • API 调用对没
    • 任务目标完成没

    这些信号就够 Reflector 做复盘了。这一点很重要：说明 ACE 更像一种自我改进框架，而不是只能靠标注数据喂出来的 prompt optimizer。

    3. 在线适配更猛：挑战集上超过榜首 GPT-4.1 系统

    在线设置下：

    • ReAct + DC：51.9%
    • ReAct + ACE：59.5%

    比 DC 高 7.6 个点，比基线高 17.1 个点。

    更亮眼的是 leaderboard 结果：

    • IBM CUGA（GPT-4.1） 平均 60.3%
    • ReAct + ACE（DeepSeek-V3.1） 平均 59.4%

    平均分基本追平。
    在更难的 test-challenge 上，ACE 甚至超过对方：

    • TGC 高 8.4%
    • SGC 高 0.7%

    这相当于用一个更小的开源模型，靠更好的上下文工程，把 production 级商业 agent 打到了同一档。

    在金融任务上，ACE 把“领域知识”优势打得更明显

    1. 金融信息抽取和公式推理都涨，尤其是 Formula

    论文测了两个金融基准：

    • FiNER：金融文档细粒度实体识别，139 类标签
    • Formula：从 XBRL 财报中抽值并做数值推理

    基础模型成绩：

    • FiNER：70.7%
    • Formula：67.5%

    离线适配、有 GT labels 时：

    • ICL：平均 69.6%
    • MIPROv2：平均 70.9%
    • GEPA：平均 72.5%
    • ACE：平均 81.9%

    ACE 比 GEPA 高 9.4 个点，比基线高 12.8 个点。

    其中最夸张的是 Formula：

    • 基线 67.5%
    • GEPA 71.5%
    • ACE 85.5%

    直接比基线高 18.0 个点。

    这说明在需要精确领域知识、规则和术语的任务里，“详细作战手册”确实比“简短万能提示词”更有效。

    2. 但论文也很诚实：没有可靠反馈时，在线适配会翻车

    这篇论文一个优点是没把方法吹成万能。

    在金融任务的无标签在线适配里，ACE 和 DC 都可能退化：

    • DC (CU) 平均从基线 69.1% 掉到 65.4%
    • ACE 平均 72.9%，虽然整体还高于基线，但在 FiNER 上掉到 67.3%，低于基线 70.7%

    作者解释得很清楚：
    如果没有可靠反馈，Reflector 和 Curator 就可能把错误经验写进上下文，导致“知识污染”。

    所以 ACE 不是“无监督万能学习器”，它更依赖高质量反馈信号。Agent 场景因为有执行结果，通常更适合它。

    不只是效果更好，ACE 还明显更省时间和钱

    1. 离线适配比 GEPA 快很多

    在 AppWorld 离线适配上：

    • GEPA 延迟：53,898 秒
    • ACE 延迟：9,517 秒

    下降 82.3%

    同时 rollout 数量：

    • GEPA：1434
    • ACE：357

    减少 75.1%

    2. 在线适配比 Dynamic Cheatsheet 便宜很多

    在 FiNER 在线适配上：

    • DC 延迟：65,104 秒
    • ACE 延迟：5,503 秒
    • 降低 91.5%

    token 成本：

    • DC：17.7 美元
    • ACE：2.9 美元
    • 降低 83.6%

    论文总体总结是：ACE 的适配延迟平均降低 86.9%。

    原因也不复杂：
    ACE 只做小块增量更新，而不是每轮都让 LLM 重写整份大上下文，所以 rollout 少、token 花费也少。

    这篇论文最值得记住的，不是一个新框架名字，而是一个方向判断

    1. 作者的核心判断：对 LLM 来说，详细上下文未必是坏事

    过去很多工作默认“提示词越精炼越高级”。但这篇论文反过来说：

    对人类来说，摘要可能更好；
    对 LLM 来说，详细、完整、可组织的上下文往往更有用。

    因为模型可以自己从长上下文里挑重点，但如果你提前把经验压缩掉了，那就再也捞不回来了。

    2. 这篇论文的主要贡献，可以浓缩成三句话

    1. 指出了上下文优化里的两个常被忽视的问题：短小偏见、上下文坍塌
    2. 提出 ACE：把上下文当成可持续进化的 playbook，而不是单段 prompt
    3. 证明了这条路不仅更准，还更便宜：agent 和金融任务都有效，尤其适合有执行反馈的自我改进系统

    3. 一句话总结

    ACE 其实是在提醒我们：
    LLM 的“学习”未必都要写进参数，也可以写进一份越来越成熟的上下文手册里。
