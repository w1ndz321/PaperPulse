    DETAILED ANALYSIS

    Meta-Harness：让模型自己改“外壳”，把手写提示工程卷成自动化搜索

    TL;DR

    大模型系统的效果，很多时候不只看模型本身，还看外面的“harness”（也就是负责存什么、取什么、怎么喂给模型的那层代码）。这篇论文发现，过去优化这层代码的方法看得太少、记得太短，像“只看分数改作业”。作者提出 Meta-Harness：让一个 coding agent 直接读所有历史代码、分数和执行日志，在文件系统里自己翻案情、改 harness。结果上，文本分类比 ACE 高 7.7 个点且上下文少 4 倍；数学推理平均提升 4.7 个点；TerminalBench-2 上还超过了多个人工精调 agent。

    过去的方法为什么不行：优化 harness 不能只看“考试分数”

    1. harness 不是提示词小修小补，而是整套“上下文调度系统”

    论文里对 harness 的定义很明确：它是包在 LLM 外面的代码，决定哪些信息要存、何时检索、怎么组织给模型看。

    这件事有多重要？作者在引言里直接说，同一个 benchmark、同一个模型，只改 harness，性能差距能到 6 倍。也就是说，很多时候不是模型不行，而是“喂法”不行。

    2. 现有文本优化方法，给的信息太压缩了

    作者批评得很直接：现在很多 text optimizer 不适合优化 harness，因为它们的反馈太“失真”了。常见问题有三类：

    • 无记忆：只看当前候选，不看完整历史
    • 只看标量分数：像只知道“这次考了 62 分”，不知道错在哪
    • 只看摘要：把长执行日志压成几句总结，细节全丢了

    论文 Table 1 很能说明问题。过去方法每次迭代能看到的上下文规模，通常只有：

    • OPRO：0.002M tokens
    • TextGrad：0.015M
    • AlphaEvolve：0.022M
    • GEPA：0.008M
    • TTT-Discover：0.026M

    而 Meta-Harness 直接拉到：

    • 10.0M tokens / iteration

    这是数量级上的差异。作者还说，在他们的设置里，单次评估最多会产生 10,000,000 tokens 的诊断信息，比之前大很多个量级。
    很好理解：优化 harness 不是改一句 prompt，而是在改一个“多步流程系统”，如果中间检索、记忆、状态更新哪个环节出问题，只看最终分数根本追不到源头。

    Meta-Harness 怎么做：把所有失败记录都放进文件系统，让 agent 自己查案

    1. 核心思路：不是喂一个总结，而是给它整个“案卷库”

    Meta-Harness 的外循环很简单，就三步：

    1. propose：agent 去文件系统里翻历史候选的代码、分数、执行轨迹，提出新 harness
    2. evaluate：跑任务，评估新 harness
    3. log：把新代码、分数、日志再存回文件系统

    然后循环。

    这里最关键的设计，不是“搜索”本身，而是让 proposer 能访问完整历史经验。每个候选 harness 都会留下一个目录，里面有：

    • 源代码
    • 评估分数
    • 执行轨迹
    • prompts
    • 工具调用
    • 模型输出
    • 状态更新

    不是把这些粗暴塞进一个超长 prompt，而是让 coding agent 像程序员一样用 grep、cat 去查。

    2. 为什么一定要 coding agent，而不是普通 LLM

    原因很现实：历史文件太多，塞不进上下文窗。
    所以 proposer 不能只是“看一段 prompt 然后续写”，而必须像开发者一样：

    • 决定该看哪些文件
    • 比较哪些旧版本
    • 检查改动是否引入回归
    • 直接改代码并验证接口是否通过

    论文里给了一个很具体的数据：在最难的设置下，proposer 每次迭代中位数会读取 82 个文件，并且会参考 20 多个历史候选。

    这就像从“只看成绩单改题”升级成“翻所有错题本、草稿纸、监考录像，再决定改哪里”。

    3. 外循环反而刻意做得很“笨”

    这篇文章一个挺有意思的点是：作者没有在外层写很多花里胡哨的搜索规则。
    Meta-Harness 只维护一个候选集合和 Pareto frontier，但不硬编码父代选择规则。agent 可以自由看任何历史 harness，不被限制必须从哪个 parent 变异。

    作者的想法是：把诊断和修改决策都交给 proposer，自身尽量少加人工启发式。这样未来 coding agent 变强，系统会自动吃到红利。

    文本分类结果最炸裂：精度更高，还把上下文成本砍到 1/4

    1. 比手工设计的 ACE 高 7.7 个点，而且上下文少 4 倍

    在在线文本分类实验里，作者用了 3 个数据集：

    • LawBench
    • Symptom2Disease
    • USPTO-50k

    最终 test set 平均结果（Table 2）：

    • Zero-shot：27.4
    • Few-shot(all)：40.8
    • MCE：40.0
    • ACE：40.9
    • Meta-Harness：48.6

    也就是说，Meta-Harness 相比最强人工 baseline ACE：

    • +7.7 个点
    • 额外上下文 token 从 50.8K 降到 11.4K

    相当于效果更好，同时更省上下文。这点很重要，因为很多系统提升其实是“拿更多上下文硬堆出来”的，但这篇不是。

    2. 对比文本优化器：10 倍更快追平，最后还多赢 10 多个点

    在同样预算下，作者对比了：

    • Best-of-N
    • OpenEvolve
    • TTT-Discover
    • GEPA

    搜索集准确率（Table 4）：

    • GEPA：Median 32.6 / Best 40.2
    • Best-of-N：34.0 / 44.2
    • OpenEvolve：39.1 / 43.3
    • TTT-Discover：34.1 / 45.6
    • Meta-Harness：50.0 / 56.7

    论文还强调：Meta-Harness 用约 0.1× 的评估次数，就能追平 OpenEvolve、TTT-Discover 的最终水平；最后收敛到的结果还高出 10 多个点。

    这说明它不只是“搜得更久”，而是搜得更对路。

    3. 真正起作用的不是分数，而是原始执行轨迹

    作者还做了一个很关键的消融实验（Table 3）：

    • Scores Only：Median 34.6，Best 41.3
    • Scores + Summary：Median 34.9，Best 38.7
    • Meta-Harness（full traces）：Median 50.0，Best 56.7

    结论很硬：
    摘要并没有补回关键信息，甚至可能因为压缩过度而更差。
    真正起决定作用的是原始 execution traces。

    4. 换 9 个没见过的数据集，依然是最强

    OOD 评测用了 9 个新数据集，平均准确率（Table 5）：

    • Zero-shot：67.0
    • Few-shot(32)：69.6
    • ACE：70.2
    • Meta-Harness：73.1

    比 ACE 再高 2.9 个点，而且在 9 个数据集里拿了 6 个第一。
    这说明它学到的不是某个数据集的小技巧，而是比较通用的分类 harness 策略。

    数学检索和代码智能体也有效：一个发现的 harness，跨 5 个模型都能涨分

    1. 数学推理：不是“加检索就行”，而是要找到对的检索策略

    作者研究的是 IMO 级别数学题的检索增强推理。难点在于：
    以前大家知道“类似题的解法可能有帮助”，但检索常常帮倒忙，因为检错了、喂错了、时机错了。

    他们给 Meta-Harness 一个包含 50 万+ 已解问题的语料库，让它自己搜“怎样检索、何时检索、给模型看什么”。

    最终在 200 道没见过的 IMO-level 问题上，跨 5 个 held-out 模型平均表现（Table 6）：

    • No Retriever：34.1
    • Dense Retrieval(k=1)：34.4
    • Dense Retrieval(k=5)：38.1
    • Random Few-shot：32.2
    • BM25 Retrieval：37.5
    • Meta-Harness：38.8

    相对无检索平均提升 4.7 个点，比 BM25 也高 1.3 个点。

    更关键的是，这个 harness 不是只对一个模型有效，而是跨下面 5 个模型都能涨：

    • GPT-5.4-nano
    • GPT-5.4-mini
    • Gemini-3.1-Flash-Lite
    • Gemini-3-Flash
    • GPT-OSS-20B

    这说明它找到的是可迁移的检索策略，不是某个模型的特供 prompt。

    2. TerminalBench-2：自动搜出来的 agent，打赢手工精调基线

    在 agentic coding 任务上，作者拿 TerminalBench-2 做实验。这是 89 个高难、长流程、全自动任务组成的 benchmark，本来就是各家 agent 拼刺刀的地方。

    结果（Table 7）：

    Claude Opus 4.6

    • Terminus-KIRA：74.7%
    • Meta-Harness：76.4%

    Claude Haiku 4.5

    • Goose：35.5%
    • Terminus-KIRA：33.7%
    • Meta-Harness：37.6%

    也就是说：

    • 在 Opus 4.6 上，Meta-Harness 超过手工设计的 Terminus-KIRA，排到 leaderboard 第 2
    • 在 Haiku 4.5 上，Meta-Harness 直接拿到所有已报告 Haiku 4.5 agent 的 第 1

    论文图 1 右图展示的就是这一点：模型自己优化出来的 harness，已经能在公开竞争 benchmark 上赢过人工调出来的 agent。

    这篇论文真正的新意：不是“自动调 prompt”，而是“自动做 harness 工程”

    1. 搜索对象变了：从文本片段，变成可执行代码

    过去很多工作优化的是 prompt、模板、短文本。
    这篇搜的是完整 harness 代码：包括提示构造、检索逻辑、记忆更新、调度编排。

    这意味着它改的不是“话术”，而是“工作流”。

    2. 反馈接口变了：从摘要和分数，变成全历史可检索经验库

    Meta-Harness 的核心创新，不是某个复杂搜索算法，而是一个非常工程、但很有效的设计：

    • 把完整历史经验都保存在文件系统
    • 让 proposer 自己按需查阅
    • 不要提前替它做过度总结

    这有点像从“领导给你一页 PPT 总结”变成“把所有原始材料都开放给你，你自己判断”。

    3. 它已经表现出“复盘—归因—修正”的能力

    论文在 TerminalBench-2 的案例里提到一个细节：早期 proposer 同时改了结构和 prompt，结果两个候选都退化；随后它主动怀疑是 prompt 改动把效果搞混了，于是把结构修改和 prompt 修改拆开验证，最后转向更安全的增量改法，并找到最佳候选。

    这说明它并不是机械试错，而是在做某种因果归因式的 debug。

    总结：当 coding agent 足够强，harness 工程可能会先于“模型训练”被自动化

    这篇论文最值得记住的一句话其实是：大模型系统的表现，不只由权重决定，也由 harness 决定。

    Meta-Harness 给出的答案很朴素：

    • 不要只看分数
    • 不要只看摘要
    • 把历史代码和日志都留下来
    • 让一个会写代码、会查文件、会复盘的 agent 自己做 harness 工程

    结果也很扎实：

    • 文本分类：+7.7 点，上下文 降到 1/4
    • 数学推理：跨 5 模型平均 +4.7 点
    • TerminalBench-2：超过手工强基线，Haiku 4.5 上 排名第 1

    它真正打开的方向是：
    以后我们可能不再手工调 agent 外壳，而是让 agent 自己不断重写自己的外壳。

    如果这个趋势继续下去，提示工程、上下文工程、工作流工程，很可能都会从“手工活”变成“搜索问题”。
