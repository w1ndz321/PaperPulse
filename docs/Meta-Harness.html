    DETAILED ANALYSIS

    Meta-Harness：让AI自己改“外挂代码”，把手写LLM系统优化成榜单第一

    TL;DR

    很多LLM应用效果不好，不一定是模型太弱，而是“模型外面的那层代码”写得不够好——比如记什么、取什么、怎么喂给模型。过去这类 harness 主要靠人手调。Meta-Harness 把这件事自动化：让一个 coding agent 直接读取历史代码、分数和执行日志，在代码空间里迭代改 harness。结果很猛：文本分类比 ACE 高 7.7 个点且上下文少 4 倍；IMO 级数学推理平均提升 4.7 个点；TerminalBench-2 上拿到 Haiku 4.5 榜首。

    这篇论文盯上的真问题：同一个模型，换套 harness 结果能差 6 倍

    1. harness 是什么？其实就是模型外面的“操作系统”

    论文里说的 harness，不是模型参数，而是包在模型外面的那层代码：
    - 存什么信息
    - 什么时候检索
    - 把哪些内容放进提示词
    - 多轮过程里怎么更新状态

    说白了，模型像发动机，harness 更像变速箱+导航+仪表盘。发动机不变，整车表现也能差很多。

    2. 过去主要靠人手调，而且很难调对

    论文开头就给了一个很扎眼的事实：同一个基座模型，仅仅换 harness，性能差距能到 6×。
    问题是，这种优化今天大多还靠人： - 看失败样例 - 猜哪里写得不对 - 改一点 prompt / memory / retrieval 逻辑 - 再跑一次

    这事既慢，又吃经验。

    3. 为什么现有“文本优化器”不太适合干这个活？

    作者认为，现有 prompt/text optimizer 的反馈太“压缩”了： - 有些只看当前候选，不看完整历史 - 有些只吃一个标量分数 - 有些只吃短模板或摘要

    但 harness 的问题往往是长链条的：前面一个“存什么”的决策，可能几步之后才出 bug。
    如果你只看分数或者一段总结，就像医生只看体温，不看化验单，很难定位病根。

    论文 Table 1 里给了个很直观的对比：
    以前方法每次迭代看到的上下文规模，大概是 0.002M 到 0.026M tokens；
    而 Meta-Harness 允许单次评估产生最高 10,000,000 tokens 的诊断信息，量级直接高了 约 3 个数量级。

    它怎么做的：不是压缩历史，而是把所有失败记录都扔进文件系统

    1. 核心想法：让 proposer 像工程师一样翻日志、改代码

    Meta-Harness 是个外层搜索系统。它不是直接让模型“想一个更好的 prompt”，而是让一个 coding agent 去做这件事。

    每轮流程很简单： 1. 读取历史候选 harness 的代码、分数、执行轨迹 2. 提出新的 harness 代码 3. 在任务上评测 4. 把新代码、分数、轨迹继续存回文件系统 5. 循环

    关键点在于：历史不是被压成一句总结，而是完整保留在文件系统里。
    agent 可以用 grep、cat 这些普通开发工具，按需翻看任何旧实验。

    2. 为啥文件系统访问这么关键？

    因为历史信息太大，根本塞不进一个 prompt。
    所以作者不让 proposer 一口吞下全部历史，而是让它“自己查”。

    论文里给了两个很具体的数据： - 在最难的设置里，proposer 每轮中位数会读取 82 个文件 - 每一步会参考 20 个以上的历史候选

    这很像一个靠谱工程师排查线上故障：不会把整个仓库一次性读完，而是带着假设去搜日志、对比版本、定位回归原因。

    3. 搜索空间也更自由：直接改完整程序，不是填模板

    Meta-Harness 搜的是 代码空间，不是几个固定 prompt 槽位。
    它能改的包括： - retrieval 逻辑 - memory 更新方式 - prompt 构造 - orchestration 流程 - 甚至整段程序重写

    论文里每个 harness 都是一个单文件 Python 程序。实验里 proposer 用的是 Claude Code + Opus-4.6，基座任务模型本身保持冻结。

    不是“多看点日志”这么简单：原始执行轨迹才是提分关键

    1. 论文做了很干净的消融实验

    作者专门比较了三种 proposer 可见信息：

    方法 看分数 看代码 看摘要 看原始轨迹 中位准确率 最佳准确率
    Scores Only ✓ ✓ × × 34.6 41.3
    Scores + Summary ✓ ✓ ✓ × 34.9 38.7
    Meta-Harness（完整） ✓ ✓ - ✓ 50.0 56.7

    结论很明确：原始 execution traces 才是真正有用的信号。
    只给分数不行；给摘要也不太行，甚至可能更差，因为摘要会把关键细节“压没了”。

    2. 作者的观点很朴素，但很重要

    他们的核心论断可以总结成一句话：

    harness 优化不是“挑分数最高的版本”，而是“顺着日志找到为什么坏，再改对地方”。

    这也是 Meta-Harness 和很多文本优化方法最大的分水岭。
    它不是在“摘要上做优化”，而是在“真实失败现场”上做优化。

    三组实验都打出了效果：文本分类、数学推理、Agent Coding 全面有效

    1. 文本分类：比 ACE 高 7.7 个点，还少用 4 倍上下文

    在线文本分类用了 3 个数据集：Law、S2D、USPTO。
    最佳结果如下：

    方法 平均准确率 额外上下文
    Zero-shot 27.4 0
    Few-shot(all) 40.8 12.3K
    MCE 40.0 28.5K
    ACE 40.9 50.8K
    Meta-Harness 48.6 11.4K

    两个关键信息： - 比 ACE 高 7.7 个点 - 上下文从 50.8K 降到 11.4K，差不多是 4× 更省

    也就是说，它不是“堆更多上下文换准确率”，而是把上下文用得更聪明。

    2. 跑得也更快：4 次评估就追平别人 60 次后的水平

    论文 Figure 1 给了很强的结论：
    在文本分类上，Meta-Harness 只用 4 次评估，就追平了次优方法最终性能；
    整体上和 OpenEvolve、TTT-Discover 相比，作者总结为：10× 更少评估就能达到类似水平，最终结果还高 10 多个点。

    这说明它不仅“能找到更好 harness”，而且搜得更快。

    3. OOD 泛化：换 9 个没见过的数据集，平均还是第一

    作者还测了 9 个搜索时完全没见过的数据集。结果： - Meta-Harness：73.1 - ACE：70.2 - Few-shot(32)：69.6

    平均领先 2.9 个点，并且在 9 个数据集里拿了 6 个第一。
    这说明它学到的不是某个数据集的小技巧，而是更通用的分类 harness 策略。

    4. IMO 级数学推理：一套检索 harness，迁移到 5 个模型都涨

    数学任务用了 20 万级别以上的题库，检索语料超过 50 万道已解问题，最终在 200 道 IMO-level 题上测试。
    重点是：搜索时只用一个模型选 harness，但测试时拿去给 5 个 held-out 模型 一起跑。

    平均结果： - No Retriever：34.1 - BM25 Retrieval：37.5 - Meta-Harness：38.8

    相对无检索，平均提升 4.7 个点；
    相对 BM25，平均再高 1.3 个点。

    更重要的是，它不像 dense retrieval 或 random few-shot 那样，在某些模型上会明显翻车；Meta-Harness 在 5 个模型上都保持正收益。

    5. TerminalBench-2：Haiku 4.5 榜首，Opus 4.6 超过手工强基线

    在 agent coding 基准 TerminalBench-2 上：

    Claude Haiku 4.5

    • Claude Code：27.5
    • Terminus-KIRA：33.7
    • Goose：35.5
    • Meta-Harness：37.6

    直接拿到 Haiku 4.5 榜首，比第二名 Goose 高 2.1 个点。

    Claude Opus 4.6

    • Terminus-KIRA：74.7
    • Meta-Harness：76.4
    • ForgeCode：81.8（作者说其公开代码无法完整复现）

    也就是说，在可比较的公开强基线里，Meta-Harness 也已经超过了人工精调系统。

    这篇论文最值得记住的，不是“又一个搜索器”，而是自动化 harness engineering 真的开始成立了

    1. 它证明了一个趋势：模型之外的系统层，也可以被端到端优化

    过去大家总盯着模型权重、训练数据、对齐方法。
    但这篇论文强调：模型外部的系统设计，同样值得被系统化搜索。

    而且一旦 coding agent 足够强，这个搜索空间就会突然变得“可进入”。
    作者甚至明确说，这种工作流大概是到 2026 年前后 coding agent 明显变强后 才真正变得实用。

    2. 论文的真正贡献，是“保留完整经验”，而不是“发明复杂外循环”

    这篇工作有点反常识：它外层算法并不复杂，甚至可以说很“朴素”。
    真正厉害的是接口设计： - 不过度手写搜索规则 - 不强行压缩反馈 - 让 agent 自己去读、查、比、改

    这有点像把“自动调 prompt”升级成了“自动做系统工程”。

    3. 局限也很明确

    论文也没把话说满： - proposer 主要用的是 Claude Code，是否对别的 coding agent 同样有效，还没系统研究 - TerminalBench-2 的搜索和最终评测用了同一套 benchmark，这更像“发现型任务”，虽然作者做了人工和 regex 检查来防止明显泄漏 - 计算成本不算低，只是相对这种工程搜索来说，作者认为几小时 wall-clock 已经可接受

    4. 一句话总结

    如果你把 LLM 应用看成“模型 + 外挂代码”的组合，那这篇论文的意义就是：

    以后不只是模型能训练，连外挂代码都能被 AI 自己迭代优化。

    而且这次不是小修小补，已经在文本分类、数学推理、Agent Coding 三个场景里，打出了可量化、可迁移、可上榜的结果。
